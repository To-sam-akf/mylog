# MyLog 开发步骤文档

## p0：项目初始化

**需求：**
初始化前后端项目结构，建立统一的工程目录，方便后续开发、调试和部署。

**核心逻辑：**
创建 `frontend` 和 `backend` 两个主目录。前端使用 Vue + Vite 初始化，后端使用 FastAPI 初始化，并提前规划好路由、组件、服务、数据库模型等目录结构。

推荐目录结构：

```text
mylog/
├── frontend/
└── backend/
```

前端目录：

```text
frontend/
├── src/
│   ├── api/
│   ├── views/
│   ├── components/
│   ├── router/
│   └── styles/
```

后端目录：

```text
backend/
├── app/
│   ├── main.py
│   ├── config.py
│   ├── database.py
│   ├── models/
│   ├── schemas/
│   ├── routers/
│   └── services/
```

**预期结果：**
项目可以正常启动：

```bash
cd frontend
npm run dev
```

```bash
cd backend
uvicorn app.main:app --reload
```

浏览器可以访问前端页面，后端可以打开 FastAPI 文档页面。

**测试策略：**

1. 前端访问 `http://localhost:5173`，确认页面正常显示。
2. 后端访问 `http://localhost:8000/docs`，确认 FastAPI 文档正常打开。
3. 检查前后端目录结构是否清晰，避免后续代码堆在单个文件中。

## p1：数据库设计与初始化

**需求：**
使用 MySQL 存储用户信息、页面内容、图片资源、页面历史版本和 Agent 对话记录。

**核心逻辑：**
先创建核心数据表：

```text
users
pages
page_versions
assets
agent_sessions
agent_messages
```

其中 `pages` 表用于存储当前阶段最重要的两个页面：

```text
life-notes
works
```

初始化时插入两条默认页面数据：

```sql
INSERT INTO pages (slug, title, content_html)
VALUES
('life-notes', 'Life Notes', '<p>记录生活。</p>'),
('works', 'Works', '<p>展示作品。</p>');
```

**预期结果：**
MySQL 中存在完整的数据表，并且 `pages` 表中已经有 `life-notes` 和 `works` 两个页面的初始内容。

**测试策略：**

1. 使用 MySQL 客户端查看表是否创建成功。
2. 执行：

```sql
SELECT * FROM pages;
```

确认返回两条记录。
3. 检查 `slug` 字段是否唯一，避免后续页面查询混乱。

## p2：后端数据库连接

**需求：**
FastAPI 后端能够连接 MySQL，并支持后续通过 ORM 或 SQL 操作数据库。

**核心逻辑：**
在 `backend/app/database.py` 中配置数据库连接。建议使用 SQLAlchemy 管理数据库模型。

配置内容包括：

```text
数据库地址
数据库端口
数据库用户名
数据库密码
数据库名称
```

建议将敏感信息写入 `.env` 文件，例如：

```env
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASSWORD=your_password
MYSQL_DATABASE=mylog
```

**预期结果：**
FastAPI 启动时可以成功连接 MySQL，不出现数据库连接错误。

**测试策略：**

1. 启动 FastAPI，确认没有连接报错。
2. 写一个临时测试接口，例如：

```text
GET /api/health/db
```

返回：

```json
{
  "database": "connected"
}
```

3. 断开数据库后重新访问接口，确认后端能正确返回错误信息。

## p3：页面读取接口开发

**需求：**
访客和管理员都可以打开 Life Notes 和 Works 页面，并读取页面内容。

**核心逻辑：**
开发公开接口：

```text
GET /api/pages/{slug}
```

其中 `slug` 可取：

```text
life-notes
works
```

后端根据 `slug` 查询 `pages` 表，返回页面标题、正文内容、更新时间等信息。

返回示例：

```json
{
  "slug": "life-notes",
  "title": "Life Notes",
  "content_html": "<p>记录生活。</p>",
  "updated_at": "2026-06-30 17:00:00"
}
```

**预期结果：**

访问：

```text
GET /api/pages/life-notes
GET /api/pages/works
```

可以分别返回 Life Notes 和 Works 的页面内容。

**测试策略：**

1. 使用 FastAPI `/docs` 测试接口。
2. 分别请求 `life-notes` 和 `works`，确认返回内容正确。
3. 请求不存在的页面，例如：

```text
GET /api/pages/test
```

应返回 404。
4. 不登录也可以访问该接口。

## p4：管理员登录功能

**需求：**
只有管理员可以编辑页面内容、上传图片，普通访客只能查看页面。

**核心逻辑：**
开发登录接口：

```text
POST /api/auth/login
```

用户提交：

```json
{
  "username": "admin",
  "password": "password"
}
```

后端验证密码后返回 JWT Token。

返回示例：

```json
{
  "access_token": "jwt-token",
  "token_type": "bearer"
}
```

后续所有管理员接口都需要携带：

```text
Authorization: Bearer <token>
```

**预期结果：**

管理员可以登录成功，并拿到 Token。
错误账号或密码无法登录。

**测试策略：**

1. 使用正确账号密码登录，确认返回 Token。
2. 使用错误密码登录，确认返回 401。
3. 使用空用户名或空密码登录，确认返回参数错误。
4. 使用 Token 请求管理员接口，确认权限通过。
5. 不带 Token 请求管理员接口，确认被拒绝。

---

## p5：当前用户状态接口

**需求：**
前端需要判断当前用户是否是管理员，从而决定是否显示编辑按钮。

**核心逻辑：**
开发接口：

```text
GET /api/auth/me
```

前端携带 Token 请求该接口。
后端解析 JWT，返回当前用户信息。

返回示例：

```json
{
  "username": "admin",
  "role": "admin"
}
```

如果未登录或 Token 无效，则返回未认证状态。

**预期结果：**
前端可以根据该接口判断：

```text
已登录管理员：显示编辑按钮
未登录访客：只显示普通页面
```

**测试策略：**

1. 携带正确 Token 请求，返回管理员信息。
2. 不携带 Token 请求，返回 401。
3. 携带伪造 Token 请求，返回 401。
4. Token 过期后请求，应返回登录失效。

---

## p6：页面保存接口开发

**需求：**
管理员可以编辑 Life Notes 和 Works 页面内容，并将修改后的内容保存到数据库。

**核心逻辑：**
开发管理员接口：

```text
PUT /api/admin/pages/{slug}
```

请求示例：

```json
{
  "title": "Life Notes",
  "content_html": "<p>今天记录一些生活内容。</p>",
  "content_json": {}
}
```

后端逻辑：

```text
1. 校验 Token。
2. 判断用户是否为 admin。
3. 根据 slug 查询页面。
4. 更新页面标题和正文。
5. 更新 updated_at。
6. 可选：写入 page_versions 表。
7. 返回保存成功。
```

**预期结果：**
管理员可以成功修改页面内容，刷新页面后内容仍然存在。
访客不能调用该接口。

**测试策略：**

1. 管理员修改 `life-notes`，保存后重新读取，确认内容变化。
2. 管理员修改 `works`，保存后重新读取，确认内容变化。
3. 不带 Token 请求保存接口，应返回 401。
4. 普通用户请求保存接口，应返回 403。
5. 请求不存在的 `slug`，应返回 404。
6. 保存空内容时，应根据规则决定是否允许。

## p7：页面历史版本功能

**需求：**
防止误删或错误编辑，管理员每次保存页面时，可以保留历史版本。

**核心逻辑：**
每次管理员保存页面内容时，在更新 `pages` 表之前，先把旧内容写入 `page_versions` 表。

保存内容包括：

```text
page_id
title
content_html
content_json
content_markdown
created_by
created_at
```

后续可以扩展版本回滚功能。

**预期结果：**
每次修改页面，数据库中都会保留一份旧版本记录。

**测试策略：**

1. 连续保存同一个页面 3 次。
2. 查询 `page_versions` 表，确认产生历史记录。
3. 检查历史版本内容是否是修改前的旧内容。
4. 确认不会因为自动保存太频繁导致版本数量暴涨。
   MVP 阶段可以只在手动保存时记录版本，自动保存不记录版本。

## p8：图片上传接口开发

**需求：**
管理员可以在 Life Notes 和 Works 中上传图片，并插入到页面正文中。

**核心逻辑：**
开发接口：

```text
POST /api/admin/upload/image
```

请求类型：

```text
multipart/form-data
```

后端处理流程：

```text
1. 校验管理员权限。
2. 检查文件类型。
3. 检查文件大小。
4. 生成随机文件名。
5. 保存到 static/uploads/。
6. 写入 assets 表。
7. 返回图片 URL。
```

支持格式：

```text
jpg
jpeg
png
webp
gif
```

建议单张图片大小限制：

```text
5MB
```

返回示例：

```json
{
  "url": "/static/uploads/xxx.png",
  "filename": "xxx.png"
}
```

**预期结果：**
管理员上传图片后，前端可以拿到图片 URL，并插入到编辑器中。

**测试策略：**

1. 上传 jpg/png/webp 图片，确认成功。
2. 上传超过大小限制的图片，确认失败。
3. 上传 `.exe`、`.js` 等非法文件，确认失败。
4. 不登录上传图片，确认返回 401。
5. 上传后访问图片 URL，确认浏览器可以正常显示。

## p14：自动保存功能

**需求：**
管理员编辑内容时，系统自动保存，减少手动保存负担。

**核心逻辑：**
前端监听编辑器内容变化，并使用防抖保存。

流程：

```text
用户输入
↓
内容发生变化
↓
等待 1000ms
↓
如果没有继续输入，调用保存接口
↓
保存成功后显示“已保存”
```

防抖时间建议：

```text
800ms - 1500ms
```

保存状态：

```text
保存中
已保存
保存失败
```

**预期结果：**
管理员编辑内容后，不需要频繁手动点击保存，页面可以自动保存最新内容。

**测试策略：**

1. 连续输入文字，确认不会每个字符都触发请求。
2. 停止输入 1 秒后，确认自动调用保存接口。
3. 保存成功后显示“已保存”。
4. 断网或后端异常时显示“保存失败”。
5. 刷新页面后确认内容已保存。

---

## p15：离开页面前保存

**需求：**
避免用户编辑后立即关闭页面或跳转页面，导致最后一部分内容没有保存。

**核心逻辑：**
监听浏览器事件：

```text
beforeunload
```

或监听路由切换事件，在离开当前页面前执行一次保存。

逻辑：

```text
如果内容有未保存变化
↓
执行最后一次保存
↓
必要时提示用户
```

**预期结果：**
用户编辑内容后直接关闭页面，也尽可能保留最新内容。

**测试策略：**

1. 编辑内容后立即刷新页面，确认数据尽量不丢失。
2. 编辑内容后切换路由，确认保存接口被调用。
3. 没有内容变化时，不应重复保存。
4. 保存失败时，前端应提示用户。

---

3. 登录后刷新页面，仍保持登录状态。
4. 点击退出登录后，编辑按钮消失。

## p17：权限联调

**需求：**
保证访客只能查看，管理员才能编辑，前后端权限逻辑一致。

**核心逻辑：**

前端控制：

```text
未登录：隐藏编辑按钮
已登录：admin 显示编辑按钮
```

后端控制：

```text
公开接口：允许所有人访问
管理员接口：必须校验 Token 和 role
```

需要重点保护的接口：

```text
PUT /api/admin/pages/{slug}
POST /api/admin/upload/image
```

**预期结果：**
访客无法通过页面或直接请求接口修改数据。

**测试策略：**

1. 未登录时前端不显示编辑按钮。
2. 未登录直接请求保存接口，应返回 401。
3. 使用错误 Token 请求保存接口，应返回 401。
4. 使用普通用户 Token 请求保存接口，应返回 403。
5. 管理员 Token 请求保存接口，应成功。

## p19：Agent 后端接口开发

**需求：**
前端聊天窗口可以向后端发送问题，并获得回答。

**核心逻辑：**
开发接口：

```text
POST /api/agent/chat
```

请求：

```json
{
  "message": "你有哪些作品？",
  "session_id": "optional-session-id"
}
```

MVP 阶段可以先返回固定回答或简单规则回答。

后续再接入知识库。

基础逻辑：

```text
1. 接收用户问题。
2. 如果没有 session_id，则创建新会话。
3. 保存用户消息。
4. 调用 agent_service 生成回答。
5. 保存 assistant 消息。
6. 返回 answer 和 session_id。
```

**预期结果：**
用户在网页右下角输入问题后，可以收到 Agent 回复。

**测试策略：**

1. 发送普通问题，确认后端返回回答。
2. 不传 `session_id`，确认后端生成新会话。
3. 传入已有 `session_id`，确认消息进入同一会话。
4. 输入空消息，应返回参数错误。
5. 前端连续发送多条消息，确认聊天记录正常显示。
