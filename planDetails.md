# MyLog 分阶段时间计划

> 本文基于 `plan.md` 拆分每一步的执行时间、阶段任务和达成效果。时间以个人开发节奏估算，可根据实际进度前后调整。

## 第 1 天上午：p0 项目初始化

**要完成的计划：**

- 建立 `frontend`、`backend` 两个主目录。
- 前端使用 Vue + Vite 初始化项目。
- 后端使用 FastAPI 初始化项目。
- 提前创建 `api`、`views`、`components`、`router`、`styles`、`models`、`schemas`、`routers`、`services` 等基础目录。
- 分别启动前端和后端开发服务。

**达到的效果：**

- 项目结构清晰，前后端职责分离。
- 前端可以访问 `http://localhost:5173`。
- 后端可以访问 `http://localhost:8000/docs`。
- 后续功能可以按模块继续开发，不会把代码堆在单个文件中。

## 第 1 天下午：p1 数据库设计与初始化

**要完成的计划：**

- 设计 MySQL 核心数据表：`users`、`pages`、`page_versions`、`assets`、`agent_sessions`、`agent_messages`。
- 创建数据库和表结构。
- 为 `pages` 表插入 `life-notes` 和 `works` 两条默认页面数据。
- 确认 `slug` 字段具备唯一性。

**达到的效果：**

- MySQL 中已经具备 MyLog MVP 所需的基础数据结构。
- `life-notes` 和 `works` 两个页面有初始内容。
- 后续接口可以直接基于真实数据表开发。

## 第 2 天上午：p2 后端数据库连接

**要完成的计划：**

- 在 `backend/app/database.py` 中配置 MySQL 连接。
- 引入 SQLmodel 或项目选定的数据库访问方式。
- 使用 `.env` 管理数据库地址、端口、用户名、密码和数据库名。
- 增加临时健康检查接口 `GET /api/health/db`。

**达到的效果：**

- FastAPI 启动时可以成功连接 MySQL。
- 数据库连接配置不硬编码在业务代码中。
- 可以通过健康检查接口判断数据库是否可用。

## 第 2 天下午：p3 页面读取接口开发

**要完成的计划：**

- 开发公开接口 `GET /api/pages/{slug}`。
- 支持读取 `life-notes` 和 `works` 页面。
- 返回 `slug`、`title`、`content_html`、`updated_at` 等字段。
- 对不存在的页面返回 404。

**达到的效果：**

- 访客和管理员都可以读取页面内容。
- 前端 Life Notes 和 Works 页面具备数据来源。
- 页面读取流程从数据库到接口形成闭环。

## 第 3 天上午：p4 管理员登录功能

**要完成的计划：**

- 开发 `POST /api/auth/login`。
- 完成管理员账号密码校验。
- 登录成功后返回 JWT Token。
- 登录失败时返回明确的 401 或参数错误。
- 为后续管理员接口准备 Token 校验逻辑。

**达到的效果：**

- 管理员可以通过账号密码获取访问 Token。
- 错误账号或密码无法登录。
- 系统初步具备身份认证能力。

## 第 3 天下午：p5 当前用户状态接口

**要完成的计划：**

- 开发 `GET /api/auth/me`。
- 解析请求中的 JWT Token。
- 返回当前用户的 `username` 和 `role`。
- 对无 Token、伪造 Token、过期 Token 返回未认证状态。

**达到的效果：**

- 前端可以判断当前用户是否为管理员。
- 页面可以根据登录状态显示或隐藏编辑入口。
- 登录态刷新后的恢复逻辑具备接口基础。

## 第 4 天上午：p6 页面保存接口开发

**要完成的计划：**

- 开发管理员接口 `PUT /api/admin/pages/{slug}`。
- 校验 Token 和管理员角色。
- 根据 `slug` 更新页面标题、正文内容和更新时间。
- 对不存在页面返回 404。
- 明确空内容是否允许保存。

**达到的效果：**

- 管理员可以修改 Life Notes 和 Works 页面内容。
- 保存后重新读取页面可以看到最新内容。
- 访客无法调用保存接口修改数据。

## 第 4 天下午：p7 页面历史版本功能

**要完成的计划：**

- 在页面保存前，将旧内容写入 `page_versions` 表。
- 记录 `page_id`、标题、HTML、JSON、Markdown、创建人和创建时间。
- MVP 阶段先只在手动保存时记录历史版本。
- 连续保存多次后检查版本记录是否正确。

**达到的效果：**

- 页面内容每次修改前都有历史备份。
- 误删或误改后具备后续恢复基础。
- 自动保存不会造成版本数量失控。

## 第 5 天上午：p8 图片上传接口开发

**要完成的计划：**

- 开发 `POST /api/admin/upload/image`。
- 支持 `multipart/form-data` 上传。
- 校验管理员权限、文件类型和文件大小。
- 将图片保存到 `static/uploads/`。
- 写入 `assets` 表并返回图片 URL。

**达到的效果：**

- 管理员可以上传 jpg、jpeg、png、webp、gif 图片。
- 非法文件和超大文件会被拒绝。
- 前端可以拿到图片 URL 并插入编辑器。

## 第 5 天下午：p9 前端路由与页面框架

**要完成的计划：**

- 配置 Vue Router。
- 创建 `/`、`/life-notes`、`/works`、`/login` 四个路由。
- 搭建 `App.vue` 全局框架。
- 增加导航栏和 Agent 悬浮按钮占位。
- 测试刷新页面后的路由表现。

**达到的效果：**

- 用户可以通过导航栏切换主要页面。
- 前端页面结构稳定。
- 后续页面开发有统一入口。

## 第 6 天上午：p10 主页开发

**要完成的计划：**

- 开发 Home 页面。
- 展示网站名称 MyLog 和个人简介。
- 增加 Life Notes、Works 两个入口。
- 点击入口跳转到对应页面。
- 检查桌面端和移动端布局。

**达到的效果：**

- 用户打开网站后能理解这是个人生活记录和作品展示网站。
- 两个核心页面入口清晰可见。
- 首页具备基本视觉完整度。

## 第 6 天下午：p11 Life Notes 前端页面开发

**要完成的计划：**

- 页面加载时请求 `GET /api/pages/life-notes`。
- 实现 `loading`、`view`、`editing`、`saving`、`saved`、`error` 状态。
- 未登录时只展示内容。
- 管理员登录后显示编辑按钮和保存状态。

**达到的效果：**

- 访客可以查看 Life Notes。
- 管理员可以进入编辑模式。
- 接口异常时页面有错误反馈。

## 第 7 天上午：p12 Works 前端页面开发

**要完成的计划：**

- 页面加载时请求 `GET /api/pages/works`。
- 复用 Life Notes 的读取和权限逻辑。
- 根据作品集场景优化页面文案和展示风格。
- 支持较长内容和项目截图展示。

**达到的效果：**

- 访客可以查看作品集。
- 管理员可以编辑 Works 页面。
- 作品内容、技术栈、项目截图等信息可以正常展示。

## 第 7 天下午：p13 富文本编辑器开发

**要完成的计划：**

- 选择 TipTap、Quill 或 WangEditor 作为富文本编辑器。
- 开发通用组件 `RichEditor.vue`。
- 支持标题、段落、加粗、列表、图片等基础能力。
- 内容变化时向父组件同步最新 HTML。
- 初步处理粘贴外部内容的安全问题。

**达到的效果：**

- Life Notes 和 Works 可以共用同一个编辑器。
- 管理员可以编辑文字和图片。
- 保存后页面格式不丢失。

## 第 8 天上午：p14 自动保存功能

**要完成的计划：**

- 监听编辑器内容变化。
- 使用 800ms 到 1500ms 防抖自动调用保存接口。
- 增加保存中、已保存、保存失败状态。
- 防止连续输入时产生过多请求。

**达到的效果：**

- 管理员编辑内容后可以自动保存。
- 停止输入一段时间后自动提交最新内容。
- 保存成功或失败都有明确提示。

## 第 8 天下午：p15 离开页面前保存

**要完成的计划：**

- 监听 `beforeunload` 或路由切换事件。
- 判断是否存在未保存变化。
- 离开前执行最后一次保存。
- 保存失败时提示用户。

**达到的效果：**

- 用户刷新、关闭或切换页面时，最后修改内容尽量不丢失。
- 没有内容变化时不会重复保存。
- 编辑体验更可靠。

## 第 9 天上午：p16 前端登录页面开发

**要完成的计划：**

- 开发 `/login` 页面。
- 调用 `POST /api/auth/login` 获取 Token。
- 将 Token 保存到 `localStorage`。
- 登录后调用 `/api/auth/me` 获取用户信息。
- 实现退出登录，清除 Token 和用户状态。

**达到的效果：**

- 管理员可以通过登录页进入编辑状态。
- 登录后刷新页面仍能保持登录态。
- 退出后恢复访客视图。

## 第 9 天下午：p17 权限联调

**要完成的计划：**

- 联调前端登录态和后端权限校验。
- 未登录时隐藏编辑按钮。
- 管理员登录后显示编辑按钮。
- 直接请求管理员接口时也必须通过后端权限校验。
- 测试无 Token、错误 Token、普通用户 Token、管理员 Token 四种情况。

**达到的效果：**

- 访客无法通过页面或接口修改数据。
- 管理员权限在前后端表现一致。
- 核心写操作具备基本安全保障。

## 第 10 天上午：p18 Agent 悬浮按钮开发

**要完成的计划：**

- 开发 `AgentFloatingButton.vue`。
- 开发 `AgentChatBox.vue`。
- 在 `App.vue` 中全站挂载。
- 实现展开、关闭、输入框、消息区、发送按钮等基础 UI。
- 检查移动端是否遮挡主要内容。

**达到的效果：**

- 用户在任意页面都能看到 Agent 悬浮按钮。
- 点击按钮可以展开聊天窗口。
- 聊天窗口具备完整的输入和展示结构。

## 第 10 天下午：p19 Agent 后端接口开发

**要完成的计划：**

- 开发 `POST /api/agent/chat`。
- 接收 `message` 和可选 `session_id`。
- 没有 `session_id` 时创建新会话。
- 保存用户消息和 assistant 回复。
- MVP 阶段先返回固定回答或简单规则回答。

**达到的效果：**

- 前端可以向后端发送问题并获得回答。
- Agent 对话具备会话记录基础。
- 后续接入知识库时有稳定接口。

## 第 11 天上午：p20 Agent 前后端联调

**要完成的计划：**

- 前端点击发送后立即展示用户消息。
- 调用 `/api/agent/chat`。
- 后端返回后展示 Agent 回复。
- 发送过程中禁用发送按钮。
- 后端异常时显示错误提示。

**达到的效果：**

- Agent 聊天窗口可以完成基础问答。
- 用户能看到发送中和失败状态。
- MVP 阶段具备可用的对话体验。

## 第 11 天下午：p21 静态资源访问配置

**要完成的计划：**

- 在 FastAPI 中挂载 `/static/uploads`。
- 确认上传文件保存路径为 `backend/app/static/uploads/`。
- 前端使用完整图片地址进行渲染。
- 检查图片 URL 不暴露服务器敏感路径。

**达到的效果：**

- 上传后的图片可以通过浏览器访问。
- Life Notes 和 Works 中插入的图片可以正常显示。
- 静态资源访问路径适合后续部署转发。

## 第 12 天上午：p22 XSS 与内容安全处理

**要完成的计划：**

- 后端保存前清洗 HTML。
- 前端渲染前进行必要防护。
- 禁止 `<script>`、事件属性、`javascript:`、`iframe`、`object`、`embed` 等危险内容。
- 保留常用富文本标签，如段落、标题、列表、图片、链接、代码块等。

**达到的效果：**

- 页面能正常展示富文本内容。
- 恶意脚本不会被保存或执行。
- 图片、标题、列表等正常内容不会被误删。

## 第 12 天下午：p23 页面样式与响应式设计

**要完成的计划：**

- 统一全站基础样式、字体、按钮、边框和内容宽度。
- 优化主页、Life Notes、Works、Login、Agent 聊天框样式。
- 适配桌面端、平板和手机端。
- 检查图片、导航栏、富文本内容和聊天框是否溢出。

**达到的效果：**

- MyLog 具备简洁、清晰、统一的视觉风格。
- 不同屏幕尺寸下都能正常阅读和操作。
- 页面不出现明显错位、遮挡或横向滚动问题。

## 第 13 天：p24 前后端整体联调

**要完成的计划：**

- 按访客视角完整浏览主页、Life Notes 和 Works。
- 按管理员视角登录、编辑页面、上传图片、自动保存。
- 刷新页面后确认数据仍然存在。
- 退出登录后确认无法编辑。
- 打开 Agent 并完成一次问答。
- 重启后端后确认数据仍然存在。

**达到的效果：**

- MyLog 核心功能形成完整闭环。
- 前端、后端、数据库、静态资源、权限和 Agent 都能协同工作。
- MVP 进入可验收状态。

## 第 14 天上午：p25 部署准备

**要完成的计划：**

- 前端执行 `npm run build`。
- 后端准备生产运行命令。
- 梳理 Nginx 转发规则：`/`、`/api/`、`/static/`。
- 检查生产环境 MySQL、环境变量、上传目录和 HTTPS 配置。
- 验证前端路由刷新不会 404。

**达到的效果：**

- 项目具备从本地迁移到服务器的条件。
- 公网部署路径、API 路径和静态资源路径清楚。
- 生产环境风险点提前暴露。

## 第 14 天下午：p26 MVP 验收标准

**要完成的计划：**

- 按访客、管理员、移动端用户、异常情况四个视角验收。
- 确认主页、Life Notes、Works 可以访问。
- 确认管理员可以登录、编辑、上传图片、自动保存。
- 确认访客不能编辑。
- 确认 Agent 可以打开并完成基础问答。

**达到的效果：**

- MyLog 第一版达到可用标准。
- 核心功能不是单点可用，而是完整流程可用。
- 可以作为基础个人网站开始使用。

## MVP 完成后：p27 后续增强方向

**要完成的计划：**

- 根据使用反馈决定下一批功能优先级。
- 可优先考虑 Works 独立作品条目、Life Notes 文章列表、版本回滚、图片资源管理。
- Agent 方向可继续接入个人知识库、流式输出、页面内容自动同步。
- 内容管理方向可增加 Markdown 导入导出、搜索、后台管理页面。
- 每新增一个功能，都重新测试权限、保存、展示、移动端、安全和部署兼容性。

**达到的效果：**

- MyLog 从 MVP 逐步演进为个人主页、生活记录、作品集和个人知识库 Agent 的组合系统。
- 后续迭代有清晰方向，不会一次性把复杂功能全部塞进 MVP。
